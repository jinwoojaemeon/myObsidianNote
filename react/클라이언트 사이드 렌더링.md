# 클라이언트 사이드 렌더링 (Client-Side Rendering, CSR)

#react #vue #spa #렌더링 #프론트엔드 #면접 #개념정리

**관련 개념:** [[서버 사이드 렌더링]] [[SPA]] [[가상 DOM]] [[번들링]] [[라우팅]]

---

## Why (왜 사용하는가? 왜 중요한가?)

- **실무**: CSR 없이는 매 페이지 이동마다 서버에 요청을 보내야 하고, 전체 페이지를 다시 로드해야 합니다. 이는 사용자 경험이 저하되고, 서버 부하가 증가하며, 인터랙티브한 웹 애플리케이션을 만들기 어렵습니다.

- **구조적 의미**: 초기 로딩 후 클라이언트에서 모든 렌더링을 처리하여 서버 부하를 줄이고, 빠른 페이지 전환과 부드러운 사용자 경험을 제공합니다. API를 통해 데이터만 주고받아 네트워크 트래픽을 최적화합니다.

- **면접 의도**: 렌더링 방식에 대한 이해, CSR과 SSR의 차이점, 각각의 장단점과 사용 시나리오를 아는지 확인하려 합니다. 최신 웹 개발 트렌드와 성능 최적화에 대한 지식을 평가합니다.

---

## Core Concept (핵심 개념 정리)

### 개념 정의

**클라이언트 사이드 렌더링(CSR)**은 브라우저(클라이언트)에서 JavaScript를 실행하여 HTML을 동적으로 생성하는 렌더링 방식입니다. 서버는 초기에 빈 HTML과 JavaScript 번들만 제공하고, 이후 모든 렌더링은 클라이언트에서 처리합니다.

> [!tip] 중요한 특징
> CSR은 SPA(Single Page Application)의 핵심 기술입니다. 초기 로딩 후 페이지 전환이 서버 요청 없이 클라이언트에서만 이루어집니다.

---

### 동작 방식

```
CSR 렌더링 흐름:

1. 사용자가 URL 접근
      ↓
2. 서버가 빈 HTML + JavaScript 번들 전송
   <div id="root"></div>
   <script src="bundle.js"></script>
      ↓
3. 브라우저가 JavaScript 다운로드 및 실행
      ↓
4. JavaScript가 가상 DOM 생성 및 렌더링
      ↓
5. API 호출로 데이터 가져오기 (필요시)
      ↓
6. 데이터로 화면 업데이트
      ↓
7. 이후 페이지 전환은 클라이언트에서만 처리
   (서버 요청 없음)
```

**관련 개념:** [[가상 DOM]] [[번들링]] [[API 호출]] [[라우팅]]

---

### 장점과 단점

> [!success] 장점
> 
> - ✅ **빠른 페이지 전환**: 서버 요청 없이 즉시 화면 전환 가능
> - ✅ **서버 부하 감소**: 렌더링 작업을 클라이언트로 분산
> - ✅ **인터랙티브한 UX**: 부드러운 애니메이션과 즉각적인 피드백
> - ✅ **서버 비용 절감**: 정적 파일만 제공하면 되므로 서버 리소스 적게 사용
> - ✅ **개발 편의성**: 프론트엔드와 백엔드 완전 분리 가능

> [!warning] 단점
> 
> - ❌ **초기 로딩 시간**: JavaScript 번들을 모두 다운로드해야 함
> - ❌ **SEO 문제**: 검색 엔진이 초기 빈 HTML만 크롤링
> - ❌ **낮은 초기 성능**: 첫 화면 표시까지 시간이 걸림
> - ❌ **JavaScript 필수**: JavaScript가 비활성화되면 작동 안 함

---

> [!info] 필요 조건
> 
> CSR을 사용하려면 다음이 필요합니다:
> - **JavaScript 프레임워크/라이브러리**: React, Vue, Angular 등
> - **번들러**: Webpack, Vite, Parcel 등
> - **라우팅 라이브러리**: React Router, Vue Router 등
> - **API 서버**: 데이터를 제공할 백엔드 API

---

### CSR vs SSR 비교

| 구분 | CSR (Client-Side Rendering) | SSR (Server-Side Rendering) |
|:---:|:---:|:---:|
| **렌더링 위치** | 브라우저 (클라이언트) | 서버 |
| **초기 HTML** | 빈 HTML + JS 번들 | 완성된 HTML |
| **초기 로딩** | 느림 (JS 다운로드 필요) | 빠름 (HTML 즉시 표시) |
| **페이지 전환** | 빠름 (서버 요청 없음) | 느림 (서버 요청 필요) |
| **서버 부하** | 낮음 (정적 파일만 제공) | 높음 (매 요청마다 렌더링) |
| **SEO** | ❌ 어려움 | ✅ 유리함 |
| **사용 사례** | 대시보드, 관리자 페이지 | 블로그, 뉴스 사이트 |

## Interview Answer Version (면접 답변식 요약)

클라이언트 사이드 렌더링은 브라우저에서 JavaScript를 실행하여 HTML을 동적으로 생성하는 방식입니다.

사용하는 이유는 빠른 페이지 전환과 부드러운 사용자 경험을 제공하기 위해서입니다. 초기 로딩 후에는 서버 요청 없이 클라이언트에서만 페이지 전환이 이루어지므로, SPA처럼 인터랙티브한 웹 애플리케이션을 만들 수 있습니다.

핵심 특징은 서버가 빈 HTML과 JavaScript 번들만 제공하고, 이후 모든 렌더링과 라우팅을 클라이언트에서 처리한다는 점입니다. 다만 초기 로딩 시간이 길고 SEO에 불리한 단점이 있어, 필요에 따라 SSR이나 SSG와 함께 사용하는 하이브리드 방식을 고려해야 합니다.

## Practical Tip (사용시 주의할 점 or 활용 예)

### 설정 시 반드시 고려해야 할 파라미터

- **코드 스플리팅**: 초기 번들 크기를 줄이기 위한 라우트별/컴포넌트별 분할
- **로딩 상태 관리**: 초기 로딩 중 사용자에게 피드백 제공
- **에러 바운더리**: JavaScript 에러 발생 시 처리
- **캐싱 전략**: 정적 자산과 API 응답 캐싱

### 흔히 발생하는 문제/오해

> [!warning] 흔한 오해
> 
> "CSR이 항상 SSR보다 빠르다"는 생각은 잘못되었습니다. 초기 로딩은 SSR이 더 빠를 수 있으며, 페이지 전환에서만 CSR이 유리합니다. 또한 SEO가 중요한 경우 CSR만으로는 부족할 수 있습니다.

### 적용 사례

- **React**: Create React App, Next.js (CSR 모드)
- **Vue**: Vue CLI, Nuxt.js (CSR 모드)
- **Angular**: Angular CLI
- **실무 예시**: 관리자 대시보드, SNS, 이커머스 상품 페이지

### 이걸 모르고 사용하면 생기는 문제

> [!error] 이걸 모르고 사용하면
> 
> - **초기 로딩 지연**: 큰 번들 크기로 인한 느린 첫 화면 표시
> - **SEO 실패**: 검색 엔진이 콘텐츠를 크롤링하지 못함
> - **메모리 누수**: 컴포넌트 언마운트 시 이벤트 리스너 정리 누락
> - **번들 크기 폭증**: 불필요한 라이브러리 포함으로 인한 성능 저하
> - **접근성 문제**: JavaScript 비활성화 시 사이트 사용 불가

## 예상 꼬리질문 정리

### 1. CSR의 초기 로딩 시간을 줄이는 방법은?

- **코드 스플리팅**: React.lazy(), dynamic import 사용
- **트리 쉐이킹**: 사용하지 않는 코드 제거
- **번들 최적화**: Webpack, Vite 등의 최적화 설정
- **CDN 활용**: 정적 자산을 CDN에서 제공
- **압축**: Gzip, Brotli 압축 적용

### 2. CSR에서 SEO 문제를 해결하는 방법은?

- **SSR 하이브리드**: Next.js, Nuxt.js 같은 프레임워크 사용
- **프리렌더링**: Prerender.io 같은 서비스 활용
- **메타 태그 동적 주입**: React Helmet 등으로 메타 정보 관리
- **서버 사이드 렌더링 전환**: 필요 시 SSR로 전환

### 3. CSR과 SSR을 함께 사용하는 경우는?

- **Next.js**: 페이지별로 CSR/SSR 선택 가능
- **Nuxt.js**: Universal 모드로 하이브리드 렌더링
- **사용 시나리오**: 
  - 공개 페이지는 SSR (SEO 중요)
  - 인증 필요 페이지는 CSR (빠른 전환)

### 4. 가상 DOM이 CSR에서 중요한 이유는?

- **효율적인 업데이트**: 변경된 부분만 실제 DOM에 반영
- **성능 최적화**: 배치 업데이트로 리플로우/리페인트 최소화
- **개발 편의성**: 선언적 UI 작성 가능
- **크로스 브라우저 호환성**: 브라우저 차이 추상화
