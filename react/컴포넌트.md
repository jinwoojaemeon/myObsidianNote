## Why (왜 사용하는가? 왜 중요한가?)

- **실무**: 컴포넌트 없이는 코드 중복이 심각해지고, 유지보수가 어려워집니다. 버그 수정 시 여러 곳을 동시에 수정해야 하며, 일관성 없는 UI가 발생합니다. 대규모 프로젝트에서는 코드 관리가 불가능해집니다.

- **구조적 의미**: UI를 독립적이고 재사용 가능한 단위로 분리하여 모듈화를 제공합니다. 관심사의 분리를 통해 각 컴포넌트가 단일 책임을 가지게 하여 코드의 가독성과 테스트 용이성을 극대화합니다.

- **면접 의도**: 개발자의 코드 설계 능력, 재사용성에 대한 이해, 그리고 대규모 프로젝트 관리 역량을 확인하려 합니다. 컴포넌트 기반 아키텍처의 이해도를 평가합니다.

---

## Core Concept (핵심 개념 정리)

### 개념 정의

**컴포넌트(Component)**는 독립적이고 재사용 가능한 UI 조각입니다. 하나의 기능이나 뷰를 담당하는 독립적인 모듈로, 자체적인 상태(State)와 속성(Props)을 가질 수 있습니다.

> [!tip] 중요한 특징
> 컴포넌트는 "레고 블록"처럼 조립하여 더 큰 UI를 만들 수 있습니다. 각 컴포넌트는 독립적으로 개발, 테스트, 재사용이 가능합니다.

---

### 동작 방식

```
컴포넌트 기반 개발 흐름:

1. 컴포넌트 설계 및 분리
   - UI를 논리적 단위로 분해
   - 재사용 가능한 부분 식별
      ↓
2. 컴포넌트 구현
   - Props로 데이터 받기
   - State로 내부 상태 관리
   - 이벤트 핸들러 정의
      ↓
3. 컴포넌트 조합
   - 부모-자식 관계 구성
   - 데이터 흐름 설계 (Props Down, Events Up)
      ↓
4. 렌더링
   - 가상 DOM 생성
   - 실제 DOM 업데이트
```

### 장점과 단점

> [!success] 장점
> 
> - ✅ **재사용성**: 한 번 작성한 컴포넌트를 여러 곳에서 사용 가능
> - ✅ **유지보수성**: 수정이 필요할 때 해당 컴포넌트만 수정하면 됨
> - ✅ **테스트 용이성**: 각 컴포넌트를 독립적으로 테스트 가능
> - ✅ **코드 가독성**: 작은 단위로 분리되어 이해하기 쉬움
> - ✅ **협업 효율성**: 여러 개발자가 동시에 다른 컴포넌트 작업 가능

> [!warning] 단점
> 
> - ❌ **초기 설계 복잡도**: 컴포넌트 분리 기준을 정하기 어려울 수 있음
> - ❌ **Props Drilling**: 깊은 계층 구조에서 Props 전달이 번거로울 수 있음
> - ❌ **과도한 분리**: 너무 작게 나누면 오히려 복잡도가 증가할 수 있음

---

> [!info] 필요 조건
> 
> 컴포넌트를 사용하려면 컴포넌트 기반 프레임워크나 라이브러리가 필요합니다:
> - **React**: 함수형 컴포넌트, 클래스 컴포넌트
> - **Vue**: Single File Component (.vue)
> - **Angular**: @Component 데코레이터
> 
> 각 프레임워크는 컴포넌트 생명주기, 상태 관리, 이벤트 처리 방식을 제공합니다.

### 컴포넌트 vs 모듈 비교

| 구분 | 컴포넌트 | 모듈 |
|:---:|:---:|:---:|
| **목적** | UI 조각을 독립적으로 관리 | 코드를 논리적으로 분리 |
| **범위** | 주로 프론트엔드 UI | 프론트엔드/백엔드 모두 |
| **재사용** | UI 재사용 중심 | 기능 재사용 중심 |
| **예시** | Button, Card, Modal | 유틸리티 함수, API 모듈 |
| **특징** | Props, State, 생명주기 | 함수, 클래스, 네임스페이스 |

## Interview Answer Version (면접 답변식 요약)

컴포넌트는 독립적이고 재사용 가능한 UI 조각입니다. 

사용하는 이유는 코드의 재사용성과 유지보수성을 높이기 위해서입니다. 예를 들어, 버튼 컴포넌트를 한 번 만들면 프로젝트 전체에서 일관된 스타일과 동작을 가진 버튼을 사용할 수 있고, 버튼에 변경사항이 생겼을 때 컴포넌트 한 곳만 수정하면 모든 곳에 반영됩니다.

핵심 특징은 Props를 통해 데이터를 받고, State로 내부 상태를 관리하며, 이벤트를 통해 부모 컴포넌트와 통신한다는 점입니다. 이를 통해 관심사의 분리를 실현하고, 대규모 프로젝트에서도 체계적으로 코드를 관리할 수 있습니다.

## Practical Tip (사용시 주의할 점 or 활용 예)

### 설정 시 반드시 고려해야 할 파라미터

- **Props 타입 정의**: TypeScript나 PropTypes를 사용하여 Props 타입을 명확히 정의
- **State 관리 전략**: 로컬 State vs 전역 State (Redux, Zustand 등) 결정
- **컴포넌트 크기**: 너무 크면 재사용 어려움, 너무 작으면 오버엔지니어링

### 흔히 발생하는 문제/오해

> [!warning] 흔한 오해
> 
> "모든 것을 컴포넌트로 만들어야 한다"는 생각은 위험합니다. 단순한 div나 span을 무작정 컴포넌트로 만들면 오히려 복잡도만 증가합니다. 재사용 가능성과 복잡도를 고려하여 적절히 분리해야 합니다.

### 적용 사례

- **React**: `function Button({ onClick, children }) { ... }`
- **Vue**: `<template><button @click="handleClick">{{ text }}</button></template>`
- **Angular**: `@Component({ selector: 'app-button', ... })`

### 이걸 모르고 사용하면 생기는 문제

> [!error] 이걸 모르고 사용하면
> 
> - Props Drilling 문제: 깊은 계층에서 Props를 계속 전달해야 하는 번거로움
> - 불필요한 리렌더링: State 변경 시 불필요한 하위 컴포넌트까지 리렌더링
> - 컴포넌트 분리 기준 미흡: 언제 분리해야 할지 모르는 상황
> - 생명주기 관리 실수: 메모리 누수나 이벤트 리스너 정리 누락

## 예상 꼬리질문 정리

### 1. 컴포넌트를 언제 분리해야 할까?

- **재사용 가능성이 있을 때**: 여러 곳에서 사용될 가능성이 있는 UI
- **복잡도가 높을 때**: 하나의 컴포넌트가 너무 많은 책임을 가질 때
- **독립적으로 테스트하고 싶을 때**: 특정 기능을 독립적으로 검증해야 할 때

### 2. Props와 State의 차이는 무엇일까?

| 구분 | Props | State |
|:---:|:---:|:---:|
| **소유권** | 부모 컴포넌트가 전달 | 컴포넌트 내부에서 관리 |
| **변경 가능성** | 읽기 전용 (Immutable) | 변경 가능 (Mutable) |
| **데이터 흐름** | 부모 → 자식 (단방향) | 컴포넌트 내부 |
| **용도** | 외부에서 받는 데이터 | 컴포넌트 내부 상태 |

### 3. 함수형 컴포넌트와 클래스 컴포넌트의 차이는?

- **함수형 컴포넌트**: 간결하고, Hooks를 사용하여 상태 관리, 성능 최적화 용이
- **클래스 컴포넌트**: 생명주기 메서드 사용, this 바인딩 필요
- **현재 트렌드**: 함수형 컴포넌트 + Hooks가 표준

### 4. 컴포넌트 간 통신 방법은?

- **Props Down**: 부모 → 자식 데이터 전달
- **Events Up**: 자식 → 부모 이벤트 전달 (콜백 함수)
- **Context API**: 깊은 계층에서 Props Drilling 방지
- **상태 관리 라이브러리**: Redux, Zustand 등으로 전역 상태 관리
